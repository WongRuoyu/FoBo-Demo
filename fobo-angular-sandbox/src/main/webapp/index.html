Section 1-2

One of the benefit you get from using FoBo module is that you have a easy up/down-grade cycle with a "single point of change", the FoBo init parameter in Lift Boot. There are two functions provided by FoBo that makes this "single point of change" fit for use.
Quick and easy up/down -grade of toolkit resources is provided by FoBo via a "single point of change" as:
  • FoBo simultaneously provides several versions of the included toolkits.
  • FoBo automatically serves minified toolkit resources in production and more easily debuggable resources otherwise.

FoBo even takes care of and provides minified resources in production and more easily debugable code in development, this is done without any need for code changes, at the same time you do not clutter your applications assets space with tookit library code that any way should not be directly manipulated. By changing a available FoBo toolkit init parameters you can quickly change the version of the front end toolkit's you are using or add another FoBo supported one.

The most common place to put global references to css and javascript resources in a Lift application is in the default.html template, this project and the example below is set up like that. If you need more flexibility you can put the css/js resource reference in the page body and add the data-lift="head" attribute to it, then when lift processes the page it will be moved into the heade, this could be useful if you have page specific resource needs.
